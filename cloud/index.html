Cloud@CNR User Guide

Environment: CNR Biomics Cloud - OpenStack

Target users: authorized users of the CNR Biomics cloud infrastructure

1. Introduction

This guide describes the main operations required to use Cloud@CNR, the OpenStack-based cloud environment of CNR Biomics. The platform provides virtual computing resources, persistent storage, virtual networking and, where enabled, GPU-based instances for scientific and data-intensive workloads.

Cloud@CNR is a private OpenStack cloud infrastructure designed to provide secure, scalable and governed computing resources for scientific, biomedical, bioinformatics and data stewardship activities.

2. Access to the platform

2.1 Access through the Horizon dashboard

Users can access the cloud environment through the OpenStack Horizon web dashboard.

  • Open a web browser.
  • Access the Horizon URL provided by the Cloud@CNR team.
  • Select the correct authentication domain, if required.
  • Log in using the credentials enabled for the platform.
Depending on the user profile and the current configuration, access may rely on local credentials, institutional credentials or federated authentication mechanisms.

2.2 Access through the command line interface

To use OpenStack from the command line, install the OpenStack client: 

pip install python-openstackclient

Download the OpenStack RC file from the Horizon dashboard and load it in your shell: 

source openrc.sh

Verify that the authentication works:

openstack token issue
openstack server list

3. Main OpenStack concepts

Term Description
Project / Tenant Isolated workspace where users, quotas and resources are managed.
Instance A virtual machine created and managed by the user.
Image Base operating system used to create a virtual machine.
Flavor Resource profile assigned to a virtual machine: vCPU, RAM and disk.
Network Virtual network used by instances for communication.
Floating IP Public IP address that can be associated with an instance.
Security Group Virtual firewall controlling inbound and outbound traffic.
Volume Persistent block storage that can be attached to an instance.

4. Creating a virtual machine

4.1 Creating an instance from the dashboard

  1. Log in to Horizon.
  2. Select the appropriate project.
  3. Go to Compute → Instances.
  4. Click Launch Instance.
  5. Enter the instance name.
  6. Select the operating system image.
  7. Select the required flavor.
  8. Select the network available for the project.
  9. Configure the SSH key pair.
  10. Launch the instance.
SSH key-based access is strongly recommended. Password-based access should be avoided unless explicitly required and authorized.

4.2 Creating an instance from the CLI

openstack server create \
  --image IMAGE_NAME \
  --flavor FLAVOR_NAME \
  --network NETWORK_NAME \
  --key-name KEY_NAME \
  VM_NAME

Check the instance status:

openstack server list
openstack server show VM_NAME

5. Accessing instances through SSH

After creating the instance, associate a Floating IP if external access is required. Then connect through SSH using the private key associated with the instance. 

ssh -i key.pem ubuntu@PUBLIC_IP

The default username depends on the selected image. Common examples are:

  • ubuntu for Ubuntu images
  • debian for Debian images
  • centos for CentOS images
  • rocky for Rocky Linux images
If SSH access fails, check that port 22 is allowed in the Security Group, that the Floating IP is correctly associated and that the private key permissions are correct. 
chmod 600 key.pem

6. Networks and Floating IPs

OpenStack networks allow communication between instances and, when configured, external access through routers and Floating IPs.

6.1 Useful network commands

openstack network list
openstack subnet list
openstack router list
openstack floating ip list

6.2 Associating a Floating IP

openstack floating ip create PUBLIC_NETWORK_NAME
openstack server add floating ip VM_NAME FLOATING_IP

To remove a Floating IP association:

openstack server remove floating ip VM_NAME FLOATING_IP

7. Security Groups

Security Groups act as virtual firewalls for instances. They define which traffic is allowed to reach the virtual machine.

7.1 Common rules

  • SSH access: TCP port 22
  • HTTP access: TCP port 80
  • HTTPS access: TCP port 443
  • ICMP access: useful for ping tests, if allowed by policy

7.2 Example CLI command

openstack security group rule create \
  --proto tcp \
  --dst-port 22 \
  --remote-ip 0.0.0.0/0 \
  SECURITY_GROUP_NAME
Avoid opening services to the whole Internet unless strictly necessary. Whenever possible, restrict access to trusted IP ranges.

8. Persistent storage: volumes

Volumes provide persistent block storage that can be attached to instances. Unlike ephemeral instance disks, volumes can survive instance deletion if they are not explicitly removed.

8.1 Creating and attaching a volume

openstack volume create --size 100 VOLUME_NAME
openstack server add volume VM_NAME VOLUME_NAME

List available volumes:

openstack volume list
Volumes are recommended for data that must persist beyond the lifecycle of a single VM.

9. GPU-enabled instances

Some Cloud@CNR projects may have access to GPU-enabled flavors. These resources are intended for workloads requiring hardware acceleration, such as machine learning, molecular modelling, image processing or GPU-accelerated bioinformatics pipelines.

To launch a GPU-enabled instance, select one of the available GPU flavors from the dashboard or specify it through the CLI: 

openstack server create \
  --image IMAGE_NAME \
  --flavor GPU_FLAVOR_NAME \
  --network NETWORK_NAME \
  --key-name KEY_NAME \
  GPU_VM_NAME
GPU availability depends on project quotas, hardware availability and administrative policies.

10. Quotas and resource limits

Each project has assigned quotas that limit the amount of resources that can be used. Typical quota limits include number of instances, vCPUs, RAM, volumes, snapshots, Floating IPs and security group rules. 

openstack quota show

If a deployment fails because of quota limits, users should contact the Cloud@CNR team and provide the project name, the requested resources and the error message.

11. Basic troubleshooting

Problem Possible checks
Instance is not reachable Check Floating IP association, Security Group rules and instance status.
SSH access fails Check key permissions, username, port 22 rule and network connectivity.
Instance creation fails Check project quotas, flavor availability, image status and network availability.
Volume cannot be attached Check volume status, availability zone and project quota.
Floating IP cannot be allocated Check Floating IP quota and public network availability.

12. Best practices

  • Use SSH keys for instance access.
  • Restrict Security Group rules to trusted networks whenever possible.
  • Use volumes for persistent data.
  • Regularly remove unused instances, volumes and Floating IPs.
  • Choose flavors according to the real needs of the workload.
  • Avoid storing sensitive data on temporary disks.
  • Document software environments and analysis pipelines for reproducibility.

13. Support

For technical support, users should contact the Cloud@CNR / CNR Biomics technical team. Support requests should include all relevant information needed to reproduce or diagnose the issue.

Please include:

  • Project name
  • Instance name or ID
  • Error message
  • Operation being performed
  • Date and approximate time of the issue
